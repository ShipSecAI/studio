## Subdomain Discovery

Find every subdomain attached to a target domain before you start scanning. More subdomains = more attack surface coverage.

### Subfinder

<Info>
  Passive subdomain enumeration using 40+ data sources — no direct interaction with the target.
  [GitHub](https://github.com/projectdiscovery/subfinder) · Docker: `ghcr.io/shipsecai/subfinder`
</Info>

| Input | Type | Description |
|---|---|---|
| `domains` | Array | The target domains you want to enumerate |
| `providerConfig` | Secret | Optional `provider-config.yaml` to unlock authenticated sources (Shodan, Censys, etc.) |

| Output | Type | Description |
|---|---|---|
| `subdomains` | Array | All discovered subdomain hostnames |
| `subdomainCount` | Number | Total count |
| `rawOutput` | String | Raw tool output for debugging |
```bash
# What runs under the hood
subfinder -d example.com -silent
```

---

### Amass

<Info>
  Deep subdomain enumeration using both passive sources and active DNS techniques.
  [GitHub](https://github.com/owasp-amass/amass) · Docker: `ghcr.io/shipsecai/amass`
</Info>

| Input | Type | Description |
|---|---|---|
| `domains` | Array | Target domains |
| `mode` | Select | `passive` — no direct contact · `active` — DNS brute-force and zone transfers |

| Parameter | Type | Description |
|---|---|---|
| `timeout` | Number | Max run time in minutes |
| `maxDns` | Number | DNS queries per second (throttle to avoid rate limits) |

<Tip>
  Use `passive` mode for stealth. Use `active` mode when you have explicit authorization and want maximum coverage.
</Tip>

### ShuffleDNS + MassDNS

<Info>
  High-speed DNS bruteforcing using a wordlist — resolves millions of potential subdomains in seconds.
  [ShuffleDNS](https://github.com/projectdiscovery/shuffledns) · [MassDNS](https://github.com/blechschmidt/massdns) · Docker: `ghcr.io/shipsecai/shuffledns-massdns`
</Info>

This is a combined image with both tools pre-installed. ShuffleDNS manages the wordlist bruteforce; MassDNS handles the high-throughput resolution.

| Input | Type | Description |
|---|---|---|
| `domains` | Array | Target domains to bruteforce against |
| `wordlist` | File | Subdomain wordlist (e.g. `subdomains-top1million.txt`) |
| `resolvers` | Array | Custom DNS resolver IPs to use |

## DNS Resolution

Once you have a list of subdomains, resolve them — find out which ones are real, what IPs they point to, and who's hosting them.

### DNSX

<Info>
  Fast, multi-threaded DNS resolution with support for every record type.
  [GitHub](https://github.com/projectdiscovery/dnsx) · Docker: `ghcr.io/shipsecai/dnsx`
</Info>

| Input | Type | Description |
|---|---|---|
| `domains` | Array | Subdomains or hostnames to resolve |
| `recordTypes` | Array | Which DNS records to fetch: `A`, `AAAA`, `CNAME`, `MX`, `NS`, `TXT`, etc. |
| `resolvers` | Array | Custom resolver IPs (e.g. `1.1.1.1:53`) |

| Parameter | Type | Description |
|---|---|---|
| `threads` | Number | How many lookups to run simultaneously (default: `100`) |
| `retryCount` | Number | Retries per failed lookup (default: `2`) |
| `rateLimit` | Number | Max requests per second |
| `showCdn` | Boolean | Tag results with their CDN provider (Cloudflare, Akamai, etc.) |
| `showAsn` | Boolean | Include ASN / ISP information in results |

| Output | Type | Description |
|---|---|---|
| `results` | Array | DNS responses grouped by record type |
| `resolvedHosts` | Array | Unique resolved hostnames |
| `rawOutput` | String | Raw JSONL output |

## HTTP Probing

Figure out which discovered hosts actually have a web server running — and capture key metadata about each one.

### HTTPx

<Info>
  Bulk HTTP probing — status codes, titles, tech stack, TLS info, and more.
  [GitHub](https://github.com/projectdiscovery/httpx) · Docker: `ghcr.io/shipsecai/httpx`
</Info>

| Input | Type | Description |
|---|---|---|
| `targets` | Array | Hostnames or URLs to probe |

| Parameter | Type | Description |
|---|---|---|
| `ports` | String | Ports to check (e.g. `"80,443,8080,8443"`) |
| `statusCodes` | String | Only return results with these status codes |
| `threads` | Number | Concurrent probe workers |
| `followRedirects` | Boolean | Follow HTTP redirects automatically |
| `tlsProbe` | Boolean | Include TLS certificate details |
| `path` | String | Append a specific path to every probe (e.g. `/admin`) |

| Output | Type | Description |
|---|---|---|
| `results` | Array | Per-host HTTP metadata (status, title, tech, headers) |
| `rawOutput` | String | Raw httpx JSON lines |
```bash
# What runs under the hood
httpx -l targets.txt -json -status-code 200,301
```

## Port Scanning

Discover which ports are open on a target — the first step to understanding what services are exposed.

### Naabu

<Info>
  Fast, reliable port scanner using SYN and CONNECT probes.
  [GitHub](https://github.com/projectdiscovery/naabu) · Docker: `ghcr.io/shipsecai/naabu`
</Info>

| Input | Type | Description |
|---|---|---|
| `targets` | Array | Hostnames or IP addresses to scan |

| Parameter | Type | Description |
|---|---|---|
| `ports` | String | Specific ports or ranges (e.g. `"22,80,443,1000-2000"`) |
| `topPorts` | Number | Scan the top N most common ports |
| `rate` | Number | Packets per second — lower this on rate-limited networks |
| `retries` | Number | Retry attempts per port |
| `enablePing` | Boolean | Ping the host first to check if it's up |

| Output | Type | Description |
|---|---|---|
| `findings` | Array | Open ports per target |
| `openPortCount` | Number | Total open ports found across all targets |
```bash
# What runs under the hood
naabu -host scanme.sh -top-ports 100
```

## Vulnerability Scanning

### Nuclei

<Info>
  Template-based vulnerability scanning — thousands of community-maintained detection rules built in.
  [GitHub](https://github.com/projectdiscovery/nuclei) · Docker: `ghcr.io/shipsecai/nuclei`
</Info>

This is a custom Nuclei image with the full **nuclei-templates** library baked in — no separate download needed.

| Input | Type | Description |
|---|---|---|
| `targets` | Array | URLs or hosts to scan |
| `templates` | Array | Specific template IDs or paths to run (leave empty to run all) |

| Parameter | Type | Description |
|---|---|---|
| `severity` | Array | Only run templates matching these severities: `critical`, `high`, `medium`, `low` |
| `rate` | Number | HTTP requests per second |
| `concurrency` | Number | Number of templates running in parallel |

| Output | Type | Description |
|---|---|---|
| `findings` | Array | All detected vulnerabilities with full detail |
| `criticalCount` | Number | Number of critical-severity findings |

<Warning>
  Only run Nuclei against systems you own or have explicit written authorization to test. Active scanning without permission is illegal in most jurisdictions.
</Warning>

## Secret Detection

### TruffleHog

<Info>
  Scans Git history, cloud storage, filesystems, and container images for leaked credentials.
  [GitHub](https://github.com/trufflesecurity/trufflehog) · Docker: `ghcr.io/shipsecai/trufflehog`
</Info>

| Input | Type | Description |
|---|---|---|
| `scanTarget` | String | A repo URL, file path, S3 bucket, or Docker image to scan |
| `scanType` | Select | `git` · `github` · `gitlab` · `s3` · `filesystem` · `docker` |

| Parameter | Type | Description |
|---|---|---|
| `onlyVerified` | Boolean | Only surface secrets that TruffleHog can actively verify as live (default: `true`) |
| `branch` | String | Limit scan to a specific Git branch |
| `sinceCommit` | String | Only scan commits after this reference — great for PR-level scanning |

| Output | Type | Description |
|---|---|---|
| `secrets` | Array | All detected secrets with type, location, and verification status |
| `verifiedCount` | Number | How many secrets are confirmed active |
| `hasVerifiedSecrets` | Boolean | `true` if any live secrets were found — use this to trigger alerts |
```bash
# What runs under the hood
trufflehog git https://github.com/org/repo --results=verified --json
```

## Notifications

### Notify

<Info>
  Send alerts to Slack, Discord, Telegram, or email at any point in your workflow.
  [GitHub](https://github.com/projectdiscovery/notify) · Docker: `ghcr.io/shipsecai/notify`
</Info>

| Input | Type | Description |
|---|---|---|
| `message` | String | The alert message to send |
| `provider` | Select | `slack` · `discord` · `telegram` · `email` |
| `config` | Secret | Provider configuration (webhook URL, API key, etc.) |

<Tip>
  Put **Notify** at the end of every workflow so your team gets a summary the moment a run completes — whether it found something or not.
</Tip>

## Full Example Pipeline

Here's a complete **attack surface discovery** workflow using security components end-to-end:
```
Manual Trigger       ← operator provides target domains
      ↓
  Subfinder          ← discovers all subdomains
      ↓
    DNSx             ← resolves subdomains to IPs
      ↓
    HTTPx            ← finds live web services
      ↓
   Nuclei            ← scans for vulnerabilities
      ↓
   Notify            ← alerts the team with findings
```
